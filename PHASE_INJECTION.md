# Custom spatiotemporal phase profile injection (epoch2d)

**Status:** implemented and validated · branch `my-epoch-mods` · commit `0f0a08f6`
**Scope:** epoch2d only (epoch3d port to follow)

## Summary

The custom laser injector in `epoch2d` can now read a **spatiotemporal phase
profile** from file, alongside the amplitude profile it already supported. At
every time step `laser%phase(i)` is updated by bilinear interpolation from the
phase file, exactly as `laser%profile(i)` is updated from the amplitude file.

This enables self-consistent injection of laser fields generated by
[LASY](https://github.com/LASY-org/lasy), whose complex envelope
`E(y,t) = |E(y,t)| · exp(i·φ(y,t))` carries both amplitude **and** phase
(wavefront curvature, Gouy phase, aberrations, propagation effects).

## Motivation

Previously the amplitude could be read from a LASY-generated `.dat` file, but the
phase was still specified analytically in the deck, e.g.

```
phase = 2.0 * pi/lambda0 * y^2 / (2.0 * RC) - gouy
```

For an ideal Gaussian this is fine, but for a propagated / experimental / non-paraxial
beam the analytic phase no longer matches the amplitude's true wavefront. Sourcing
the phase from the **same** envelope as the amplitude removes this inconsistency.
(For the near-paraxial Gaussian used in validation the phase contribution turned
out to be modest — ~0.5%, see Validation — but the feature is essential for
tightly focused or aberrated beams, and for any pointwise pipeline comparison.)

## Deck interface

```
begin:laser
  boundary = x_min
  intensity_w_cm2 = 1e22
  lambda = 1.0 * micron
  pol = 0

  use_custom_profile = T
  use_spatiotemporal_profile = T
  profile_data_file = amplitude_profile.dat

  use_phase_from_file = T
  phase_data_file = phase_profile.dat
end:laser
```

- `use_phase_from_file = T` — read the phase from file each time step.
- `phase_data_file` — phase filename. Relative paths resolve from the deck's
  data directory; absolute paths are used as-is. If omitted, defaults to
  `phase_profile.dat`.
- When `use_phase_from_file = T`, any `phase = ...` deck expression is **ignored**.
- Backward compatible: if `use_phase_from_file` is not set, behaviour is exactly
  as before.

## File format

Plain-text, identical to the amplitude file — only the matrix values differ:

```
Line 1:   n_t  n_y                       (two integers, n_t first)
Line 2:   y_0  y_1  ...  y_{n_y-1}       [metres]
Line 3:   t_0  t_1  ...  t_{n_t-1}       [seconds]
Lines 4+: n_t rows, each of n_y values   [radians]   ! row j corresponds to t_j
```

The phase **must lie on the same grid as the amplitude file** (same `n_y`, `n_t`,
same coordinates, uniform spacing). Whichever profile loads first establishes the
shared grid; the second validates its dimensions and EPOCH aborts with a clear
message if they differ. Uniform spacing is assumed by the O(1) index lookup
(generate with `np.linspace`).

## EPOCH field and phase convention

EPOCH injects (see `laser.f90`, `outflow_bcs_*`):

```
field  ∝  amp · t_profile(t) · profile(y,t) · sin( ω·t + phase(y,t) )
```

with `ω·t` (the carrier, `current_integral_phase = omega * time`) supplied
**internally** by EPOCH. Key points:

- `phase` is in **radians** and is used **as-is** — EPOCH applies no degree
  conversion and no constant (e.g. π/2) offset.
- Because the carrier is added internally, the file must contain **only the
  slowly varying envelope phase**, not the carrier.

Base conversion from a LASY envelope, where the physical field is
`E = Re[E·exp(−iω₀t)] = |E|·cos(ω₀t − φ)`:

```
profile(y,t) = |E(y,t)| / max|E|          (normalised amplitude envelope)
phase(y,t)   = −φ(y,t) + π/2              (π/2 converts cos→sin; sign flip for the
                                           opposite phase-argument convention)
```

The Python/LASY-side converter applies this and writes the result; EPOCH reads it
unchanged. **But see the next section — the raw `−φ + π/2` is not sufficient on
its own.**

## ⚠ Gotcha: the arbitrary global envelope phase (root cause of an early 137% error)

During validation, the full-LASY run initially disagreed with the analytic-phase
reference by a **maximum relative difference of 137%**. Root cause and fix:

**Root cause.** LASY's angular-spectrum propagator imprints an *arbitrary global
phase* (a "piston") on the envelope. After propagating from focus back to the
boundary, the on-axis envelope came out ≈ negative-real:
`angle(E) ≈ −3.112 rad ≈ −π` at the pulse peak, not 0. That global phase is
physically meaningless for an isolated beam (only the *relative* wavefront shape
matters), but it is a real number sitting in the array.

The naive conversion `phase = −angle(E) + π/2` faithfully copied that ≈ −π into
EPOCH's phase:

- buggy LASY on-axis phase: `−(−3.112) + π/2 = +4.683 rad`
- reference (deck) on-axis phase: `−gouy = −0.088 rad`
- constant difference between the two runs: `Δ ≈ 4.77 rad`

**Why that gives 137%.** EPOCH injects `sin(ω·t + phase)`. Two runs whose phase
differs by a constant `Δ` differ pointwise by `sin(θ + Δ) − sin(θ)`, whose peak
amplitude is `2·|sin(Δ/2)|`:

```
2·|sin(4.77 / 2)| = 2 × 0.686 = 1.37   ⇒   137%
```

× peak field 86 a₀ ≈ 118 a₀ — exactly the measured `max|Δ| = 118.0 a₀`. The bug is
fully accounted for.

**The fix (Python side).** Reference the LASY phase to a fixed point so the
propagator's piston cancels, and pin it to the deck convention:

```
phase_epoch = −( φ(y,t) − φ(0, t_peak) ) + (−gouy)
```

This keeps the physical wavefront curvature but removes the arbitrary global
offset. After it, the full-LASY error dropped **137% → 1.98%**.

**How to avoid it in future.** The absolute/global phase of any envelope is
arbitrary — every propagator adds its own piston. Always subtract a reference
phase (on-axis, at peak time) before writing the file. For a single standalone
run it would not even matter (a many-cycle pulse cannot observe a global phase),
but the moment you (a) compare two pipelines pointwise, or (b) do CEP-sensitive
few-cycle physics, the reference must be fixed.

(Note: a *constant* phase offset such as the π/2 cannot by itself cause a large
spatial/temporal error — it just shifts the global carrier phase. The 137% came
from the constant `Δ` differing between the two pipelines, which the reference
subtraction eliminates.)

## Validation

A controlled 2×2 experiment (amplitude source × phase source) against a
`numerical` reference (analytic amplitude + deck phase):

| Run | Amplitude | Phase | Max rel. diff |
|-----|-----------|-------|---------------|
| sanity (built-in `gauss`) | analytic | deck | 0.02% |
| isolate amplitude | LASY | deck | 1.92% |
| isolate phase | numerical | LASY file | **0.49%** |
| full LASY | LASY | LASY file | 1.98% |

The full-LASY error ≈ √(1.92² + 0.49²) — the two contributions add in quadrature.
The **phase-from-file path is correct** (phase-only error 0.49%, with the 137%
piston bug above fixed). The residual ~1.9% is an **amplitude** difference —
LASY's angular-spectrum-propagated profile vs the paraxial analytic Gaussian,
concentrated at the beam edges — **not** a phase error. (This corrected an
earlier hypothesis that the residual was phase-dominated; the "oscillatory
on-axis difference" was an amplitude error modulated by the carrier.) For tightly
focused or aberrated beams, LASY is the more accurate amplitude source.

## Implementation

All files under `epoch2d/src/`:

| File | Change |
|------|--------|
| `shared_data.F90` | `use_phase_from_file` (logical) and `phase_data_file` (string) added to `laser_block`. |
| `user_interaction/custom_laser.f90` | `file_phase_matrix` storage; `load_phase_profile` loader; `custom_laser_phase` bilinear interpolator; shared-grid load factored into `ensure_shared_grid` (so amplitude and phase reuse `file_y_coords`/`file_t_coords` and are robust to load order — phase is updated before profile in the BCs). |
| `laser.f90` | `laser_update_phase` calls `custom_laser_phase` (cell-centre `y(i)`/`x(i)`) when phase comes from file; the phase-update trigger fires for `use_phase_from_file` in all four outflow BCs. |
| `deck/deck_laser_block.f90` | parse and initialise `use_phase_from_file` and `phase_data_file`. |

Builds cleanly with `make COMPILER=gfortran` (`-std=f2003`).

## Scope and future work

- **epoch3d port** — apply the same change once 2d is fully validated.
- **Binary file format** — a binary `.dat` upgrade is planned separately; the
  current text format is retained for now.
- **Phase wrapping** — `custom_laser_phase` interpolates the phase linearly. If a
  future profile has large wavefront curvature that wraps by 2π between adjacent
  grid nodes, the phase should be unwrapped before writing (a smooth 2π offset
  does not change the final `sin`, so unwrapping is safe). Not an issue for the
  validated Gaussian case.
